# 프로젝트 설명서 — KIS 모의투자 초단기 모멘텀 돌파 자동매매

이 문서는 코드를 실행하는 방법(README.md)이 아니라, **무엇을 왜 이렇게 설계했는지**를
설명하기 위한 문서입니다.

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|---|---|
| 목적 | 한국투자증권(KIS) Open API **모의투자** 환경에서 동작하는 REST 기반 자동매매 시스템 구현 |
| 통신 방식 | REST API만 사용 (웹소켓 미사용 — 폴링 기반) |
| 거래 환경 | 모의투자 전용 (실전 도메인 미사용) |
| 인증 정보 관리 | 환경변수(`.env`)로만 로드, 코드에 하드코딩하지 않음 |
| 핵심 제약 | 모의투자는 API 호출 제한이 엄격하므로, 불필요한 호출을 최소화하는 것이 설계의 핵심 축 |

REST/폴링 방식을 선택한 이유는 단순히 "웹소켓을 안 써도 돼서"가 아니라, **이 전략 자체가
하루 30분(09:00~09:30)만 동작하는 짧은 시간대 전략**이라 실시간 스트리밍이 주는 이점(지속적인
저지연 체결 감시)이 크지 않다고 판단했기 때문입니다. 대신 그 시간 동안 호출 횟수를 더 아끼는
방향으로 설계했습니다.

## 2. 전체 아키텍처

```
main.py  (실행 진입점)
  │
  └─ trader.py — BreakoutScalpTrader (전략 + 메인 루프)
       │
       ├─ market_data.py — MarketData   (시세조회)
       ├─ account.py     — Account      (잔고조회)
       ├─ orders.py      — Orders       (매수/매도 주문)
       │        │
       │        └─ 공통으로 사용 ↓
       └─ api_client.py  — ApiClient    (HTTP 요청 공통 처리)
                │
                └─ auth.py — 토큰 발급/캐싱, hashkey 발급

config.py   — 모든 모듈이 참조하는 설정값 (환경변수, 전략 파라미터, tr_id)
logger.py   — 모든 모듈이 공유하는 로깅 채널
```

**계층을 나눈 이유**: "전략이 무엇을 할지 결정하는 부분(trader.py)"과 "API를 어떻게 호출하는지
(api_client.py 이하)"를 분리했습니다. 이렇게 하면:
- tr_id나 응답 필드명이 바뀌어도 `trader.py`(전략 로직)는 손댈 필요가 없음
- 전략을 바꾸고 싶을 때도 API 호출 코드는 그대로 재사용 가능
- 테스트할 때 API 레이어만 따로 검증 가능 (`test_connection.py`가 그 예시)

## 3. 모듈별 상세 설명

### 3.1 `config.py` — 설정 단일 진입점
- `python-dotenv`로 `.env` 파일을 읽어 `GH_ACCOUNT`, `GH_APPKEY`, `GH_APPSECRET`을 로드
- `validate_credentials()`: 환경변수가 비어있으면 즉시 `sys.exit()` — 잘못된 빈 값으로
  조용히 진행되다가 나중에 알 수 없는 에러로 죽는 것을 방지
- 계좌번호를 `CANO`(8자리) / `ACNT_PRDT_CD`(2자리)로 분리 — KIS API가 이 두 필드를
  따로 요구하기 때문
- 전략 파라미터(시간대, 보유시간, 목표수익/손실률 등)와 tr_id 상수를 한곳에 모아둠
  → 전략을 튜닝할 때 이 파일 하나만 보면 됨
- `class TrId`: 확신도가 낮은 tr_id 값을 별도로 모아 "확인 필요" 주석을 달아둠
  (이유는 6절 "한계점" 참고)

### 3.2 `logger.py` — 로깅 채널
- `RotatingFileHandler`로 `logs/trader.log`에 기록 (용량 제한 2MB, 백업 3개) +
  콘솔에도 동시 출력
- 모든 모듈이 `from logger import logger` 한 줄로 동일한 로거를 공유 → 로그 형식 통일,
  중복 핸들러 등록 방지 로직 포함

### 3.3 `auth.py` — 인증
- `get_access_token()`: `token_cache.json`에 **오늘 날짜로 발급된 토큰**이 있으면
  그대로 재사용하고, 없을 때만 `POST /oauth2/tokenP`로 신규 발급
  - 캐싱 기준은 "발급 후 경과 시간"이 아니라 **"오늘 발급된 것인가"** — 토큰 유효기간이
    약 24시간이라는 점과 모의투자 토큰 재발급 자체에도 제한이 있다는 점을 고려한 설계
- `get_hashkey()`: 주문 body의 무결성을 검증하기 위해 KIS가 요구하는 해시값을 발급받는
  함수. 주문 API에서만 사용 (시세조회/잔고조회에는 불필요)

### 3.4 `api_client.py` — 공통 HTTP 클라이언트
- 모든 API 호출에 공통으로 필요한 헤더(인증 토큰, appkey/secret, tr_id, custtype)를
  자동으로 붙여주는 단일 진입점
- 타임아웃(5초) + 재시도(최대 2회, 2초 간격) 로직을 **여기 한 곳에서만** 구현 →
  market_data/account/orders 코드는 네트워크 에러 처리를 신경 쓸 필요가 없음
  (실제로 운영 중 모의투자 서버가 간헐적으로 500 에러를 낸 적이 있는데, 이 재시도
  로직으로 자동 복구된 사례가 있었음)

### 3.5 `market_data.py` — 시세조회
- `get_current_price(symbol)` 단일 함수만 제공
- KIS의 `inquire-price` 엔드포인트를 호출해 현재가(`stck_prpr` 필드)를 정수로 반환

### 3.6 `account.py` — 잔고조회
- `get_balances()`가 **보유종목 딕셔너리**와 **가용현금**을 한 번의 호출로 같이 반환
  → 두 정보를 따로 조회하면 호출이 2배가 되므로 통합
- 보유수량이 0인 종목은 결과에서 제외 (불필요한 데이터 정리)

### 3.7 `orders.py` — 주문
- `buy_market(symbol, qty)` / `sell_market(symbol, qty)` 두 함수만 제공
- **지정가 주문은 만들지 않음** — 이 전략은 "신호가 뜨면 즉시 체결"이 핵심이라, 지정가는
  체결이 안 될 위험이 있어 전략 의도와 맞지 않다고 판단
- 주문 요청에는 `get_hashkey()`로 만든 해시값을 자동으로 포함

### 3.8 `trader.py` — 전략 로직 (가장 중요한 모듈, 4절에서 상세 설명)
- `BreakoutScalpTrader` 클래스 하나가 전략의 상태(범위 기록, 보유 포지션)와
  행동(진입 판단, 청산 판단, 메인 루프)을 모두 담당
- `RangeInfo`, `Position` — 가벼운 `dataclass`로 상태를 표현 (딕셔너리 대신 타입이
  명확한 구조체 사용)

### 3.9 `main.py` — 진입점
- `validate_credentials()` → `BreakoutScalpTrader().run()` 호출이 전부
- `Ctrl+C`(KeyboardInterrupt)를 받아 깔끔하게 종료 로그를 남기고 끝남

### 3.10 `test_connection.py` — 검증용 보조 스크립트
- 거래 시간과 무관하게 인증/시세조회/잔고조회 3가지가 정상인지 즉시 확인하는 스크립트
- 실제 자동매매 전 "API 연결이 살아있는가"를 빠르게 점검하기 위해 별도로 분리
  (메인 전략 코드를 건드리지 않고 진단 가능)

## 4. 매매 전략 상세 설명

### 4.1 전략 컨셉 — "초반 변동성 돌파 & 초단기 모멘텀"
장이 열린 직후 5~10분은 하루 중 거래량과 변동성이 가장 크게 몰리는 구간입니다. 이 시간대의
가격 방향성을 추종해서 짧게 진입하고 짧게 빠져나오는 전략입니다. 장시간 모니터링이 필요 없고,
하루 중 가장 신호가 명확한 구간에만 집중한다는 점이 이 전략의 핵심 장점입니다.

### 4.2 시간대별 단계 (`trader.py`의 `run()` 메서드가 이 분기를 그대로 구현)

| 시간 | 단계 | 동작 |
|---|---|---|
| 09:00 이전 | 대기 | 30초 간격으로 대기만 함 (`IDLE_POLL_SEC`) |
| 09:00 ~ 09:05 | 범위 기록 | 종목별 현재가를 20초 간격으로 폴링하며 실제 고가/저가를 갱신 (`record_range`) |
| 09:05 ~ 09:20 | 진입 구간 | 18초 간격으로 돌파 매수 감시 + 청산 조건도 동시에 점검 (`check_breakout_entries`, `check_exits`) |
| 09:20 ~ 09:30 | 청산 전용 구간 | 신규 진입은 막고, 보유 중인 포지션의 청산 조건만 계속 점검 |
| 09:30 이후 | 강제 청산 | 남아있는 포지션을 조건과 무관하게 전량 시장가 매도 후 프로그램 종료 |

### 4.3 매수(진입) 조건
- 09:00~09:05 동안 기록한 **고가를 현재가가 위로 돌파하는 순간** 즉시 시장가 매수
- 한 종목당 한 번만 진입 (이미 포지션이 있으면 재진입하지 않음 — `if symbol in self.positions: continue`)
- 매수 수량 = `종목당 배정 금액 // 현재가` (최소 1주 보장)

### 4.4 매도(청산) 조건 — 4가지 중 하나라도 만족하면 즉시 시장가 매도
1. 진입 후 **3분(180초) 경과** (`HOLD_SECONDS`)
2. **+0.3% 수익** 도달 (`PROFIT_TARGET_PCT`)
3. **-0.3% 손실** 도달 (`LOSS_LIMIT_PCT`)
4. **09:30 도달** — 위 조건과 무관하게 강제 청산 (`force=True`)

세 가지(시간/수익/손실) 조건은 매 폴링 주기(18초)마다 함께 점검되며, 가장 먼저 만족된
조건이 청산 이유로 로그에 기록됩니다.

### 4.5 자금관리
- 가용 현금을 **유니버스 종목 수로 균등 분배** (`available_cash * 0.9 / 종목수`)
- `0.9`를 곱하는 이유: 체결가 변동이나 수수료로 예수금이 모자라는 상황을 막기 위한
  10% 안전 마진 (`CASH_SAFETY_MARGIN`)
- 잔고조회를 매번 하지 않고, **프로그램 실행 중 처음 한 번만** 조회해서 배정 금액을 고정
  (`_ensure_allocation`이 `self.allocation_cash`가 이미 있으면 재조회하지 않음)

### 4.6 대상 종목 선정
대형주(삼성전자/SK하이닉스/현대차)에 더해, 상대적으로 변동성이 큰 코스닥 테마 종목
(에코프로비엠, 한미반도체, 알테오젠)을 추가해 총 6종목으로 구성했습니다. 돌파 전략은
변동성이 클수록 신호가 더 뚜렷하게 발생하는 경향이 있어 테스트 목적에 맞는 종목군으로
선정했습니다. (※ 실제 투자 추천이 아니라 전략 테스트용 예시 종목입니다.)

## 5. 설계 철학 — 왜 이렇게 만들었는가

### 5.1 API 호출을 최소화하기 위한 장치들
- 토큰은 **하루에 한 번만** 발급하고 캐싱 (`auth.py`)
- 잔고조회는 "자금 배정 시 1회" + "주문 1건당 체결확인 1회"로 제한
- 종목 수가 늘어난 만큼 폴링 간격도 같이 늘림 (3종목 12~15초 → 6종목 18~20초)
- 지정가 주문을 쓰지 않음으로써 정정/취소 같은 추가 호출 자체를 없앰

### 5.2 안전장치
- 모든 청산 조건과 무관하게 **09:30에는 무조건 전량 청산** — 장중 변수와 무관하게
  하루를 반드시 현금으로 마감
- 진입은 종목당 1회로 제한 — 같은 종목에 반복적으로 물량을 늘리는(피라미딩) 위험 차단
- 시장가 주문만 사용 — 즉시 체결을 보장해 "주문이 안 걸려서 리스크가 방치되는" 상황 방지

### 5.3 에러 처리
- `api_client.py`에 타임아웃/재시도를 한 곳에 모아 구현 → 일시적인 네트워크 문제나
  모의투자 서버의 간헐적 500 에러에도 전체 프로그램이 죽지 않고 복구를 시도
- 개별 함수(`record_range`, `check_breakout_entries`, `check_exits`)는 종목 하나가
  실패해도 `try/except`로 감싸 다음 종목 처리를 계속 진행 (한 종목의 오류가 전체 루프를
  멈추지 않도록)

|---|---|---|

| 잔고조회 tr_id/필드명 | ✅ 실제 호출로 확인됨 | 동일 |

| 청산 타이밍 정밀도 | 폴링 간격(18초) 단위로만 점검 | "정확히 3분"이 아니라 ±18초 오차 존재 |
| 진입가 기준 | 신호 감지 시점의 조회가를 entry_price로 사용 | 실제 체결가와 다를 수 있어(슬리피지) ±0.3% 판정이 약간 어긋날 수 있음 |
| 동시 전략 운용 | 단일 전략만 지원 | 여러 전략을 동시에 운용하려면 전략별 포지션 관리 구조 추가 필요 |
| 호출 제한 대응 | 폴링 간격으로만 조절 | 실제 KIS의 정확한 초당/분당 제한 수치는 공식 문서로 별도 확인 필요 |

## 7. Q&A

**Q. 왜 웹소켓을 안 썼나요?**
이 전략이 동작하는 시간이 하루 30분으로 짧고, 그 안에서도 폴링 간격(18~20초) 수준의
감시로 충분하다고 판단했습니다. 또한 과제 요구사항 자체가 REST 전용이었습니다.

**Q. 모의투자 호출 제한을 어떻게 고려했나요?**
토큰 캐싱, 잔고조회 호출 최소화, 종목 수에 비례한 폴링 간격 조정, 지정가 주문 미사용
(정정/취소 호출 제거) 네 가지 방향으로 호출량을 줄였습니다.

**Q. 전략 파라미터(3분, ±0.3%)는 왜 이 값으로 정했나요?**
장 초반 변동성이 짧고 강하게 나타나는 구간(보통 몇 분 내)을 빠르게 포착하고 빠져나오기
위한 값으로, 보수적인 손익 목표를 설정해 리스크를 짧게 제한하는 것을 목표로 했습니다.
